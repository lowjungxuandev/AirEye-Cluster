# Issues

Consolidated reference for deployment issues encountered during cluster
bootstrap, plus troubleshooting steps for the live system.

> Issues #2 (cert-manager webhook) and #6 (grim-app TLS rate limit) are
> **historical** — cert-manager + Let's Encrypt were removed in favor of
> Cloudflare proxy + Origin Certs (see [cloudflare-proxy.md](cloudflare-proxy.md)).
> They're kept below for context.

## Deployment Issues

Issues observed while bringing up the cluster from this repository.

### Summary

The cluster came up, but a few ordering and configuration issues needed manual
intervention during bootstrap:

- Existing one-shot Jobs could not be updated with `kubectl apply`.
- cert-manager was applied before its webhook was ready (historical — cert-manager removed).
- ArgoCD's ApplicationSet CRD was too large for client-side apply.
- Keycloak health probes targeted the wrong port.
- Vault Secrets Operator could not overwrite bootstrap-created Secrets.
- The `grim-app` TLS certificate hit a Let's Encrypt rate limit (historical — replaced by Cloudflare Origin Certs).
- ArgoCD self-heal reverted local-only fixes from GitHub `main`.
- MinIO OIDC login failed with an invalid `redirect_uri`.
- ArgoCD OIDC login failed with invalid client credentials.
- Vault UI OIDC login could not fetch the configured role.

### Issues And Fixes

| Issue | Symptom | Cause | Fix |
|-------|---------|-------|-----|
| Existing Jobs were immutable | `Job.batch ... spec.template: field is immutable` for `keycloak-bootstrap` and `redis-smoke-test` | Kubernetes Job pod templates cannot be patched after creation | Delete and recreate the Jobs before re-applying root manifests |
| cert-manager webhook was not ready *(historical)* | `failed calling webhook "webhook.cert-manager.io": connect: connection refused` | `ClusterIssuer` was applied before `cert-manager-webhook` was serving | No longer applies — cert-manager removed; TLS now via Cloudflare Origin Certs ([cloudflare-proxy.md](cloudflare-proxy.md)) |
| ArgoCD ApplicationSet CRD failed client-side apply | `metadata.annotations: Too long` for `applicationsets.argoproj.io` | The upstream CRD exceeds kubectl's client-side apply annotation limit | Apply ArgoCD with server-side apply |
| Keycloak rollout stuck | New Keycloak pod stayed unready and restarted; probes returned `404` | Health endpoints were not enabled and probes were pointed at port `8080` instead of Keycloak's management port | Set `KC_HEALTH_ENABLED=true` and use port `9000` for `/health/ready` and `/health/live` |
| VSO could not sync pre-created Secrets | `not the owner of the destination Secret` and missing owner labels in `VaultStaticSecret` events | Bootstrap Secrets were created manually, but VSO destination did not allow overwrite | Set `spec.destination.overwrite: true` for `server-secret` and `grim-app-secret` |
| `grim-app` TLS not ready *(historical)* | `grim-app-tls` stayed `Ready=False` | Let's Encrypt returned exact-identifier rate limit for `api.lowjungxuan.dpdns.org` | No longer applies — Cloudflare Origin Cert is installed manually and valid for 15 years ([cloudflare-proxy.md](cloudflare-proxy.md)) |
| ArgoCD reverted local fixes | Keycloak probes and VSO overwrite settings reverted after ArgoCD synced | ArgoCD was syncing GitHub `main`, which did not include the local fixes | Pause auto-sync or commit and push the manifest fixes to `main` |
| MinIO OIDC redirect rejected | MinIO login failed with `Invalid parameter: redirect_uri` | The Keycloak `minio` client only allowed a narrow callback URI | Update the Keycloak bootstrap job to set broader MinIO redirect URIs for the console and S3 hosts |
| ArgoCD OIDC token exchange failed | ArgoCD reported `failed to get token: oauth2: "unauthorized_client" "Invalid client or Invalid client credentials"` | The Keycloak `argocd` client credentials and the ArgoCD secret generated by VSO could drift or lack ArgoCD labels | Update the Keycloak client on every bootstrap run, and label/overwrite the VSO-generated `argocd-keycloak-oidc` Secret |
| Vault UI OIDC role fetch denied | Vault reported `Authentication failed: Error fetching role: permission denied` | Vault OIDC auth role/policy/tune settings were manual and could be missing after bootstrap | Add `vault-auth-config` Job to configure the OIDC auth method, `keycloak` role, policy, and unauthenticated auth-method listing |

### Commands Used

Recreate immutable Jobs:

```sh
kubectl -n infra delete job keycloak-bootstrap redis-smoke-test --ignore-not-found
kubectl apply -k .
```

Install ArgoCD CRDs with server-side apply when client-side apply fails:

```sh
kubectl apply --server-side=true --force-conflicts -k argocd
kubectl apply -k argocd/applications
```

Force VSO to resync after Vault is seeded:

```sh
kubectl -n infra annotate vaultstaticsecret server-secret \
  vso.hashicorp.com/force-sync="$(date +%s)" --overwrite
kubectl -n infra annotate vaultstaticsecret grim-app-secret \
  vso.hashicorp.com/force-sync="$(date +%s)" --overwrite
```

Pause ArgoCD auto-sync while applying local-only fixes:

```sh
kubectl -n argocd patch application grim-k8s --type=merge \
  -p '{"spec":{"syncPolicy":{"automated":null}}}'
```

Re-run Keycloak client bootstrap after OIDC redirect/client changes:

```sh
kubectl -n infra delete job keycloak-bootstrap --ignore-not-found
kubectl apply -k .
```

Apply Vault auth configuration after Vault is initialized and unsealed:

```sh
kubectl kustomize --enable-helm vault | kubectl apply -f -
kubectl -n infra wait --for=condition=complete job/vault-auth-config --timeout=180s
```

## Troubleshooting

Read-only commands that don't change cluster state. The destructive remediation
is called out where relevant.

### Cheat sheet

```sh
kubectl get pods,ingress,certificate,vaultstaticsecret,application -A
kubectl -n infra  get vaultstaticsecret
kubectl -n argocd get vaultstaticsecret
kubectl -n infra  get secret server-secret grim-app-secret
kubectl -n argocd get application grim-k8s
```

Or run `bash scripts/troubleshoot-secrets.sh` to print the full diagnostic
bundle in one shot.

---

### Vault edit doesn't reach the pod

1. **Is VSO syncing?**
   ```sh
   kubectl -n infra describe vaultstaticsecret grim-app-secret
   ```
   Look at the `Status.Conditions` block. `SecretSynced=True` means the K8s
   Secret was last written successfully.
2. **Force sync now:**
   ```sh
   kubectl -n infra annotate vaultstaticsecret grim-app-secret \
     vso.hashicorp.com/force-sync="$(date +%s)" --overwrite
   ```
3. **Did the K8s Secret actually change?**
   ```sh
   kubectl -n infra get secret grim-app-secret -o jsonpath='{.metadata.resourceVersion}'
   ```
   The number should bump after a real change.
4. **Did VSO trigger a rollout?**
   ```sh
   kubectl -n infra rollout history deployment grim-app
   kubectl -n vault-secrets-operator logs deploy/vault-secrets-operator-controller-manager --tail=50
   ```
5. **Manual restart:**
   ```sh
   kubectl -n infra rollout restart deployment grim-app
   ```

### VaultStaticSecret stuck `SecretSynced=False`

Common causes:

| Symptom (`kubectl describe`)                                  | Fix                                                   |
|---------------------------------------------------------------|-------------------------------------------------------|
| `VaultClientError: permission denied`                          | The Vault Kubernetes auth role `vso-grim-k8s` is not bound to the VSO ServiceAccount or lacks the `grim-k8s-read` policy. Re-run the manual Vault auth setup (see `docs/architecture.md`). |
| `VaultClientError: ... no role found`                          | The Vault role `vso-grim-k8s` is missing entirely. Create it (see `docs/architecture.md`). |
| `Vault path not found`                                         | Vault path is missing — re-run the manual seed (see `docs/architecture.md` § Vault config). |
| `connection refused`                                           | Vault is sealed or pod restarting — check `vault status`. |
| `VaultConnection not found` / `VaultAuth not found`            | The CRs in `vault-secrets-operator/` haven't been applied yet. |

### Vault is sealed

```sh
kubectl exec -n infra vault-0 -- vault status
```

If `Sealed: true`, the auto-unseal CronJob runs every minute. Check it ran
recently:

```sh
kubectl -n infra get cronjob vault-auto-unseal
kubectl -n infra get jobs -l job-name --field-selector status.successful=1 --sort-by=.status.completionTime | tail -5
```

To unseal manually:

```sh
UNSEAL_KEY=$(kubectl get secret -n infra vault-bootstrap-secret \
  -o jsonpath='{.data.VAULT_UNSEAL_KEY}' | base64 -d)
kubectl exec -n infra vault-0 -- vault operator unseal "$UNSEAL_KEY"
```

### ArgoCD says `OutOfSync` on a Secret

VSO mutates `/data` on each Secret it manages, which would normally show as
drift. The grim-k8s Application has `ignoreDifferences` on
`kind: Secret, jsonPointers: [/data]` — if a Secret still appears OutOfSync,
check that the path lives under that Application (and not another one) and
that the JSON pointer matches.

### ArgoCD says `OutOfSync` (general)

```sh
kubectl -n argocd get application grim-k8s -o yaml | yq '.status.sync, .status.health'
kubectl -n argocd logs deploy/argocd-repo-server --tail=200
```

Common causes:

- **`must specify --enable-helm`** — `kustomize.buildOptions: --enable-helm`
  must be in `argocd-cm` (not `argocd-cmd-params-cm`). See
  `argocd/patches/config.yaml`.
- **VSO CRDs missing** — happens before the `vault-secrets-operator` Application
  (sync-wave `-1`) finishes installing the chart. Wait, or re-sync that
  Application first.

### ArgoCD login → `WRONGPASS`

Symptom: Keycloak OIDC login redirects back with a Redis password error.

Cause: ArgoCD pods cached the old Redis password before VSO synced the new
one. The `argocd-redis` VaultStaticSecret lists ArgoCD's server,
repo-server and application-controller in `rolloutRestartTargets`, so this
should self-heal. If it doesn't, restart manually:

```sh
kubectl -n argocd rollout restart deploy/argocd-server deploy/argocd-repo-server
kubectl -n argocd rollout restart statefulset/argocd-application-controller
```

### Pod still uses old env after Secret update

Pod env vars are baked at start time. Confirm:

1. The workload is listed in the VaultStaticSecret's `rolloutRestartTargets`.
2. The VSO operator pod is running:
   ```sh
   kubectl -n vault-secrets-operator get pods
   ```
3. Force a manual rollout:
   ```sh
   kubectl -n infra rollout restart deployment grim-app
   ```

## Historical: grim-app-tls Let's Encrypt rate limit

`grim-app-tls` previously stuck in `False` / `Issuing` after burning the
"5 duplicate certs per 168h" Let's Encrypt limit on
`api.lowjungxuan.dpdns.org`. Root cause was that orange-clouded hosts
made HTTP-01 unreachable from outside, so every retry consumed quota.

**Resolution:** cert-manager + Let's Encrypt removed entirely. TLS is
now a manually-installed Cloudflare Origin Cert (15-year validity).
See [cloudflare-proxy.md](cloudflare-proxy.md).
